## 4.1 Before you start ⏱️ 3 min

Crossplane on its own knows nothing about any particular API. **Providers** extend Crossplane with support for a specific set of resources — AWS, GCP, Helm, Kubernetes, and so on. See [Cheatsheet §1.3](./cheatsheet#core-objects) for the definitions.

In this module you'll install [`provider-kubernetes`](https://github.com/crossplane-contrib/provider-kubernetes), the provider that lets Crossplane create any Kubernetes object in any target cluster. Then you'll create one **managed resource (MR)** directly — a `ConfigMap` — and watch Crossplane reconcile it.

Why a single MR before the XRD + Composition work in module 5? Because composition sits on top of MRs. You'll understand what a Composition *does* much faster if you've already seen one MR reconcile by itself.

You're about to: install a provider, grant it permissions, give it a ProviderConfig, and apply one MR.

## 4.2 Install the provider ⏱️ 8 min

### 1. Apply the `Provider` manifest

```bash
kubectl apply -f - <<'EOF'
apiVersion: pkg.crossplane.io/v1
kind: Provider
metadata:
  name: provider-kubernetes
spec:
  package: xpkg.upbound.io/crossplane-contrib/provider-kubernetes:v1.2.1
EOF
```

Crossplane pulls the package image and starts a provider Pod in `crossplane-system`. This takes 30–60 seconds.

```bash
kubectl get provider.pkg.crossplane.io
```

Expected output:

```
NAME                  INSTALLED   HEALTHY   PACKAGE
provider-kubernetes   True        True      xpkg.upbound.io/crossplane-contrib/provider-kubernetes:v1.2.1
```

### 2. Grant the provider permission to create resources

`provider-kubernetes` runs as its own ServiceAccount in `crossplane-system` with no permissions by default. Bind it to `cluster-admin` so it can create whatever you compose in later modules:

```bash
SA=$(kubectl get sa -n crossplane-system -o name | grep provider-kubernetes | cut -d/ -f2)
kubectl create clusterrolebinding provider-kubernetes-admin \
  --clusterrole=cluster-admin \
  --serviceaccount=crossplane-system:$SA
```

:::warning cluster-admin in a workshop cluster
You're binding `cluster-admin` on purpose. In modules 5 and 6 you'll create new kinds of resources, and a narrower role would force you to keep widening permissions — a detour that isn't what this workshop teaches. Your cluster is throwaway and isolated, so the blast radius is zero. **In a real cluster, scope this down.**
:::

### 3. Apply a `ProviderConfig`

A [ProviderConfig](./cheatsheet#core-objects) tells `provider-kubernetes` which Kubernetes API to talk to. The simplest — and the right choice for acting on the same cluster — is `InjectedIdentity`: the provider uses its own in-cluster ServiceAccount token.

```bash
kubectl apply -f - <<'EOF'
apiVersion: kubernetes.crossplane.io/v1alpha1
kind: ProviderConfig
metadata:
  name: default
spec:
  credentials:
    source: InjectedIdentity
EOF
```

The name `default` matters: Crossplane uses a ProviderConfig named `default` when an MR doesn't specify one, and the XRs in modules 5 and 6 rely on that.

<ValidateCheck check="provider-kubernetes-installed" />

When the tile turns green, the provider is installed, permissioned, and configured. Move on to your first MR.

## 4.3 Your first managed resource ⏱️ 7 min

### 1. What you're about to create

You're going to create one `Object` — provider-kubernetes' managed resource. Each `Object` wraps one Kubernetes resource the provider will create in the target cluster. In this case the target is this same cluster (thanks to the `default` ProviderConfig), and the wrapped resource is a `ConfigMap`.

When you apply the `Object`, the provider reads your spec, creates the `ConfigMap`, and reports `Ready=True`. That loop — spec in, condition out — is the shape of every Crossplane managed resource, for every provider.

### 2. Apply the MR

```bash
kubectl apply -f - <<'EOF'
apiVersion: kubernetes.crossplane.io/v1alpha2
kind: Object
metadata:
  name: hello-configmap
spec:
  forProvider:
    manifest:
      apiVersion: v1
      kind: ConfigMap
      metadata:
        name: hello-from-crossplane
        namespace: default
      data:
        message: "This ConfigMap was created by a Crossplane Object MR."
  providerConfigRef:
    name: default
EOF
```

### 3. Watch it reconcile

```bash
kubectl get object.kubernetes.crossplane.io
```

Expected output:

```
NAME              KIND        PROVIDERCONFIG   SYNCED   READY   AGE
hello-configmap   ConfigMap   default          True     True    20s
```

And the wrapped resource now exists in the cluster:

```bash
kubectl get configmap hello-from-crossplane -n default -o jsonpath='{.data.message}'
```

Expected output:

```
This ConfigMap was created by a Crossplane Object MR.
```

## 4.4 What just happened

You installed a provider, gave it credentials, and asked it to manage one resource. Crossplane now owns the `ConfigMap`: delete it with plain `kubectl delete configmap` and provider-kubernetes will recreate it within seconds to match the `Object` spec. Delete the `Object` and the `ConfigMap` goes with it.

That's the managed-resource contract. In module 5 you'll use it to compose many MRs behind one high-level `Application` XR.
