# Running the workshop on your own laptop

The main workshop runs on a shared cluster where each pair gets an isolated
[vCluster](https://www.vcluster.com) sandbox. If you just want to walk through
modules 1–4 on your own — outside the live workshop, with no vCluster, no
ArgoCD, no shared infrastructure — this page is the full recipe. Everything
runs in a single [k3d](https://k3d.io) cluster on your machine.

You do not need to clone this repo. Every manifest you need is fetched from
public URLs, every image is pulled from public GHCR tags.

## Prerequisites

| Tool | Install hint |
|------|--------------|
| [Docker](https://docs.docker.com/get-docker/) | `brew install --cask docker` on macOS |
| [kubectl](https://kubernetes.io/docs/tasks/tools/) | `brew install kubernetes-cli` |
| [helm](https://helm.sh/docs/intro/install/) | `brew install helm` |
| [k3d](https://k3d.io/#installation) | `brew install k3d` |

That's it. Crossplane itself is not a prerequisite — installing it is module 1.

## 1. Create the cluster

```bash
k3d cluster create crossplane-workshop-solo -p "8080:80@loadbalancer"
```

The `-p "8080:80@loadbalancer"` flag forwards your host's port 8080 to the
cluster's ServiceLB, which is where Envoy Gateway will terminate traffic.
When the cluster is up, `kubectl cluster-info` should show a
`k3d-crossplane-workshop-solo` context.

## 2. Install Envoy Gateway

The workshop's routing is [Gateway API](https://gateway-api.sigs.k8s.io)-native;
it does not use Ingress. Install the Envoy Gateway controller:

```bash
helm install eg oci://docker.io/envoyproxy/gateway-helm \
  --version v1.4.0 \
  --namespace envoy-gateway-system --create-namespace \
  --wait
```

## 3. Install the docs + wall + validator

A single `kubectl apply` installs the Docusaurus site (this one), the
validator sidecar, a `Gateway` listener on port 80, and the HTTPRoutes that
expose `/` → docs, `/team/local/` → the frontend you'll create in module 3,
and `/team/local/api/` → its backend:

```bash
kubectl apply -f https://raw.githubusercontent.com/ricCap/crossplane-workshop/main/gitops/solo/all.yaml
```

Wait for the docs pod to come up:

```bash
kubectl -n docs wait --for=condition=Available deploy/docs --timeout=180s
```

## 4. Open the workshop

Visit [http://localhost:8080/](http://localhost:8080/). You should land on
the same page you're reading now, but served from your k3d cluster rather
than the internet. Open the [wall](/wall) in a second tab — it will show a
single tile for the synthetic `local` pair, with all checks red until you
complete modules 1–3.

## 5. Do the modules

From here, follow modules 1–4 in order. There are two differences from the
workshop-day flow:

- **No `vcluster connect` step.** You're already on the cluster where the
  modules run (`kubectl config current-context` is
  `k3d-crossplane-workshop-solo`).
- **Your pair ID is `local`.** The validator reports a single synthetic
  pair called `local`, and the wall's tile at `/team/local/` is wired to
  the frontend + backend Services that module 3's `Application` claim
  creates in the `default` namespace.

Each module's validation button will click-through against your k3d cluster
and flip green as you progress.

## 6. Clean up

```bash
k3d cluster delete crossplane-workshop-solo
```

## What's intentionally missing

- **vCluster isolation.** You are both operator and participant; there is
  no sandbox between you and the cluster. If you `kubectl delete ns
  crossplane-system`, you break your own setup.
- **ArgoCD / GitOps.** The solo path applies raw YAML. No sync loop, no
  self-heal — if you want changes to stick, re-apply the manifests.
- **The "gotcha" reveal.** In the full workshop, the vcluster you use is
  itself produced by a Crossplane Composition. You won't see that moment
  here — Crossplane isn't managing your cluster, you are.
- **TLS.** Traffic is plain HTTP on localhost:8080.

## Troubleshooting

**`k3d cluster create` fails with "port 8080 already allocated"**
: Another process is bound to 8080 on your host. Either stop it, or pick a
  different host port (e.g. `-p "18080:80@loadbalancer"` and then visit
  `http://localhost:18080/`). Remember to use the same port in your browser.

**`helm install eg …` fails with "cannot pull OCI chart"**
: Docker Hub rate-limits anonymous OCI pulls. Authenticate with
  `docker login` (any free Docker Hub account works) and retry.

**Wall shows no tiles / `/api/pairs` returns an empty array**
: The `VALIDATOR_SOLO` env var isn't set. If you applied the manifest above,
  this should not happen; re-check with
  `kubectl -n docs get deploy docs -o yaml | grep -A1 VALIDATOR_SOLO`.

**`/team/local/` returns 404 before you've finished module 3**
: Expected. The `frontend` and `backend` Services don't exist yet — the
  `Application` claim in module 3 creates them. Once the claim goes Ready,
  the route starts working with no further action on your part.
