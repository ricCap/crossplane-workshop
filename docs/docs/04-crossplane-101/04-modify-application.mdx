## 6.1 Before you start ⏱️ 3m

So far your Composition is static: every `Application` produces the same HTML layout, the same fonts, the same page structure. In this module you'll edit the Composition and watch your tile update **without** re-applying the XR.

That's the point of composition: the abstraction (`Application`) survives, the implementation (the resources behind it) changes. Platform teams do this all the time — adding sidecar containers, swapping base images, tightening security defaults — and consumers never notice.

You're about to: edit the Composition in place, see the downstream ConfigMap update, refresh the tile.

## 6.2 Reshape the frontend ⏱️ 15m

### 1. Open the Composition for editing

```bash
kubectl edit composition applications.workshop.example.io
```

Find the `frontend-configmap` resource (the first one under `spec.pipeline[0].input.resources`) and modify the HTML inside `base.data.index.html`.

### 2. Pick at least one change

Any of these work, pick the one you like:

- **Add a subtitle** below the existing `<div id="title">`:
  ```html
  <div id="title">Loading...</div>
  <div id="subtitle" style="font-size: 1rem; color: #6b7280;">hello from my XRD</div>
  ```
- **Change the background color:**
  ```css
  body { background: #fef3c7; font-family: sans-serif; margin: 0; padding: 20px; text-align: center; }
  ```
- **Add emoji, a border, a gradient** — anything inline CSS can do inside one ConfigMap.

Save and exit the editor.

### 3. Watch Crossplane reconcile

Saving the editor is the only action you take. Everything else is Crossplane reacting to the change:

1. The API server stores the new Composition.
2. Crossplane notices the Composition has changed and re-runs the pipeline for **every** `Application` XR that points at it — in this workshop, that's your one XR; in a real platform it could be hundreds.
3. The function emits the same `frontend` ConfigMap as before, but with your new HTML inside.
4. Crossplane diffs that against the live ConfigMap and patches it in place. Your `Application` XR is never touched — its `spec` is identical to what you applied in the previous module.

Confirm the live ConfigMap now carries your edit:

```bash
kubectl get configmap frontend -n default -o jsonpath='{.data.index\.html}' | head -20
```

You should see your new HTML in the output.

### 4. Make nginx pick up the change

nginx reads `/usr/share/nginx/html/index.html` on every request, but the ConfigMap projection inside the Pod only refreshes every ~30 seconds (kubelet's periodic sync). Bounce the frontend Pod to see your change immediately:

```bash
kubectl rollout restart deployment frontend
```

### 5. Refresh the tile

Open the [workshop wall](/wall) in a new tab and click **Refresh**. Your tile should show the new HTML / CSS you edited. Your `Application` XR is still `wall-tile`, untouched — the change came from reshaping the Composition.

<ValidateCheck check="application-ready" />

The check is still green because none of the conditions it watches have changed — the Application is still Ready. That's the point.

## 6.3 What just happened

Think of the Composition as the kitchen and the XR as the order ticket. You edited the kitchen — different ingredients, different presentation — and every existing order automatically came out the new way, without anyone reissuing a ticket. The participant who applied the `Application` didn't touch a thing; their resource hasn't changed; their pods still come up Ready.

That gap between "what the user asked for" and "what the platform produces" is the whole reason platform teams use Crossplane. Concretely, three things are true after your edit:

- **The XR's `spec` is unchanged.** `kubectl get application wall-tile -n default -o yaml` returns the same YAML you applied in module 5. The contract with the consumer is untouched.
- **The composed resources are different.** The `frontend` ConfigMap now carries new HTML. If you had thirty `Application` XRs in the cluster, all thirty would have been patched the moment you saved.
- **The change is permanent for new XRs too.** Any `Application` someone applies from now on will be rendered with the new HTML — the Composition is the single source of truth for the recipe.

In a real platform this is how you roll out security patches, new defaults, observability sidecars, or cost-saving changes — all invisible to the teams using the API. The consumer's `kubectl apply` line never changes; you change what that line produces.

### Go deeper

- **Stretch: add a field.** Add `subtitle` to the XRD (with a default), patch it through the Composition into a new ConfigMap key or a backend env var, and apply a fresh XR that sets it. Good sandbox for the last ten minutes.
- [Composition revisions](https://docs.crossplane.io/latest/concepts/composition-revisions/) — how Crossplane keeps old Compositions around so existing XRs stay stable while new ones pick up the new recipe.
- [Composition functions catalogue](https://github.com/crossplane-contrib) — `function-kcl`, `function-go-templating`, `function-auto-ready` — patterns for moving beyond patch-and-transform once you need more logic.
