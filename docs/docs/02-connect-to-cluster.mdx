## 2.1 Before you start ⏱️ 2m

`kubectl` talks to *a* cluster — whichever cluster is described by the **kubeconfig** your shell currently points at. For the next five modules, every command needs to land in your workshop cluster, not in some other cluster you might have configured on your machine.

A kubeconfig is just a YAML file with the cluster's address, a certificate, and credentials. You're about to download yours, tell your shell to use it, and prove it works by applying a one-line Pod manifest.

You're about to: download a kubeconfig, export it, and apply a `hello` pod.

## 2.2 Load your kubeconfig ⏱️ 5m

### 1. Download the kubeconfig file

Your instructor has given you a QR code on the Miro board. Scan it on your laptop (or click the link the instructor posts) and download `<pair-id>.zip` — the kubeconfig is wrapped in a password-protected ZIP so a leaked link alone doesn't grant cluster access.

### 2. Unzip with the operator-shared password

Your instructor will share the ZIP password verbally or on the live Miro board. Extract the kubeconfig YAML out of the archive — the password is the same for every pair.

- **macOS / Linux:**
  ```bash
  cd ~/Downloads
  unzip <pair-id>.zip
  # prompts for the password, then writes <pair-id>.yaml next to the zip
  ```
- **Windows:** right-click `<pair-id>.zip` → **Extract All…** → enter the password the operator shared → confirm. You'll end up with `<pair-id>.yaml` in the same folder.

### 3. Point your shell at it

```bash
export KUBECONFIG=$PWD/<pair-id>.yaml
kubectl config current-context
```

(On Windows / PowerShell: `$env:KUBECONFIG = "$PWD\<pair-id>.yaml"`.)

Expected output:

```
workshop-<pair-id>
```

The `workshop-` prefix is your visual cue that you're on the right cluster. If you see `docker-desktop`, `minikube`, `k3d-…`, or anything that doesn't start with `workshop-`, the `export` didn't take effect in this shell — re-run it in the same terminal where you'll run the rest of the workshop. If `kubectl` prints an error, re-check the file path and that the unzip step actually produced the YAML.

:::tip The `export` only lives in this terminal
Open a new tab or window and the variable is gone — `kubectl` falls back to whichever cluster it was talking to before (often `docker-desktop`). Two simple ways out:

- **Re-run the export in each new terminal.** Copy-paste the same line. Fine for a 3-hour workshop.
- **Persist with one line in your shell startup file:**
  ```bash
  mkdir -p ~/.kube && mv <pair-id>.yaml ~/.kube/workshop.yaml
  echo 'export KUBECONFIG=$HOME/.kube/workshop.yaml' >> ~/.zshrc   # or ~/.bashrc
  ```
  New terminals pick it up automatically. Once the workshop is over, delete that line and `rm ~/.kube/workshop.yaml`.

<details>
<summary>Already use <code>kubectl</code> with other clusters and don't want to lose them?</summary>

Don't overwrite `~/.kube/config` with the workshop file — that wipes out every other context. Don't `cat <pair-id>.yaml >> ~/.kube/config` either: a kubeconfig is a single YAML document, not a stream, so appending two of them produces duplicate top-level keys and `kubectl` can't parse the result.

Merge them with `kubectl`'s built-in support:

```bash
KUBECONFIG=$HOME/.kube/config:$PWD/<pair-id>.yaml \
  kubectl config view --flatten --raw > ~/.kube/config.new \
  && mv ~/.kube/config.new ~/.kube/config \
  && chmod 600 ~/.kube/config \
  && kubectl config use-context workshop-<pair-id>
```

The `mv` does replace `~/.kube/config`, but it's safe: the previous line reads *both* files (that's what `KUBECONFIG=A:B` does — `kubectl` merges them in memory) and writes the union to `config.new`, so the file you're moving into place already contains every cluster, context, and user that was in your old `~/.kube/config`, plus the workshop entries. `kubectx` will now list `workshop-<pair-id>` alongside your normal contexts. After the workshop, remove it:

```bash
kubectl config delete-context workshop-<pair-id>
kubectl config delete-cluster workshop-<pair-id>
kubectl config delete-user    workshop-<pair-id>
```

</details>
:::

### 4. List namespaces as a smoke test

```bash
kubectl get namespaces
```

You should see the standard system namespaces (`default`, `kube-system`, `kube-public`, …). If you see a permission error, check you're using your own pair's kubeconfig — each pair only has access to their own cluster.

### 5. (Optional) Install handy tools

These are not required but they'll make the rest of the workshop faster. Skip if you're tight on time.

```bash
brew install kubectx k9s helm      # macOS / Linuxbrew
```

Windows / plain Linux install links are in the [Cheatsheet](./cheatsheet#12-tools-worth-installing).

### 6. (Strongly recommended) Show the cluster in your shell prompt

The single most effective way to avoid running `kubectl apply` or `helm install` against the wrong cluster is to make the active context visible in your shell prompt. Install [`kube-ps1`](https://github.com/jonmosco/kube-ps1) (Bash/Zsh) or enable [Starship](https://starship.rs)'s `kubernetes` module — both render the current context next to every prompt. When you see `(workshop-<pair-id>)` you're safe; when you see `(docker-desktop)` you know to fix it before typing. Install pointers are in the [Cheatsheet](./cheatsheet#12-tools-worth-installing).

## 2.3 Smoke test — apply a `hello` pod ⏱️ 3m {#23-smoke-test-apply-a-hello-pod}

Apply a one-container Pod so the validator has something concrete to confirm your `kubectl apply` pipeline actually writes to the cluster.

```bash
kubectl apply -f - <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: hello
  namespace: default
spec:
  containers:
    - name: hello
      image: public.ecr.aws/docker/library/busybox:1.36
      command: ["sh", "-c", "echo hello && sleep 3600"]
EOF
```

Wait a few seconds for the image pull, then confirm:

```bash
kubectl get pod hello -n default
```

Expected output:

```
NAME    READY   STATUS    RESTARTS   AGE
hello   1/1     Running   0          15s
```

<ValidateCheck check="hello-pod" />

When the tile turns green, your kubeconfig is good and `kubectl apply` works end-to-end.

## 2.4 What just happened

You pointed your terminal at a remote Kubernetes cluster and asked it to run a tiny container — and it did. The kubeconfig file is the address book: it tells `kubectl` which cluster to talk to, who you are, and how to authenticate. The `hello` Pod is the smallest possible "is this thing on?" — one container, no networking, no storage — and the fact that it reached `Running` proves the whole chain works.

You told your shell which cluster to talk to and proved the connection by writing a resource that the validator can read back. Everything from module 3 onward assumes this works.
