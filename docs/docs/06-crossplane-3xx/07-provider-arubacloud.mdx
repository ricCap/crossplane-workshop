## 7.1 Before you start ⏱️ 4m

You've now installed Crossplane providers against your own AWS / GCP / Azure accounts. **`provider-arubacloud`** is different: it points at **your platform team's shared Aruba project** with credentials they manage. You don't sign up for Aruba; you don't see the API key; you walk in and create resources.

This is the production pattern: a platform team curates the cloud surface — one shared account, one credential, opinionated guardrails — and developers consume it without seeing the secret. Your platform team has wrapped that with five Kyverno guardrails enforced at admission time:

- Only `Database`, `ContainerRegistry`, and `BlockStorage` Kinds may be created.
- Names must start with `<your-pair-id>-` so two pairs sharing the project don't collide.
- DBaaS engine, flavor, location, and storage size are clamped to a workshop-sized tier.
- Pods cannot reference the platform's Aruba credentials Secret.
- The shared workshop VPC is read-only — your DBaaS connects into it, but you cannot delete it.

You're about to: verify the provider is installed in your workshop cluster, get four shared workshop values from your instructor, create one Aruba MySQL DBaaS, run `SELECT 1` against it from a Pod, and clean up.

## 7.2 Verify your platform's wiring ⏱️ 3m

Confirm the provider is up. The platform team's tooling has installed `provider-arubacloud`, applied a `ProviderConfig` referencing a Secret they injected for you, and configured the Kyverno policies described above.

```bash
kubectl get provider.pkg.crossplane.io provider-arubacloud
```

Expected output:

```
NAME                  INSTALLED   HEALTHY   PACKAGE
provider-arubacloud   True        True      ghcr.io/riccap/provider-arubacloud:v0.3.0-workshop
```

If this check stays red after a few seconds, your pair's Aruba access hasn't been turned on yet — flag your instructor. Their dashboard has a green-light toggle for it.

The Kyverno policies the platform team ships should also be present. Confirm:

```bash

```

## 7.3 Provision a MySQL DBaaS ⏱️ 13m

The headline resource. You'll create one **DBaaS** managed resource (a real MySQL instance) using four shared workshop network values your platform team has dropped into a ConfigMap inside your cluster.

### 1. Get the shared values

Read the ConfigMap your platform team provisioned in your cluster's `default` namespace:

```bash
kubectl get configmap workshop-aruba-shared -o yaml
```

Expected output:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: workshop-aruba-shared
  namespace: default
data:
  projectId: <workshop-project-id>          # 24-character hex string
  vpcUri: <vpc-uri>                         # URI of the shared workshop VPC
  subnetUri: <subnet-uri>                   # URI of a subnet inside that VPC
  securityGroupUri: <security-group-uri>    # URI of a security group on the VPC
```

Keep the four values handy; you'll paste them in §7.3.2. (`vpcUriRef` / `subnetUriRef` / `securityGroupUriRef` on a `Database` MR don't yet support `valueFrom: configMapKeyRef` — Crossplane's `*Ref` suffix is for cross-MR references, not arbitrary ConfigMap lookups — so for now this is read-and-paste rather than read-and-bind.)

### 2. Apply the DBaaS MR

```bash
kubectl apply -f - <<'EOF'
apiVersion: dbaas.arubacloud.crossplane.io/v1alpha1
kind: DBaaS
metadata:
  name: <your-pair-id>-mysql
spec:
  forProvider:
    name: <your-pair-id>-mysql
    projectId: <workshop-project-id>
    engineId: mysql-8.0
    flavor: DBO2A4
    location: ITBG-Bergamo
    zone: ITBG-3
    billingPeriod: Hour
    storage:
      sizeGb: 20
    network:
      vpcUriRef: <vpc-uri>
      subnetUriRef: <subnet-uri>
      securityGroupUriRef: <security-group-uri>
    tags:
      - crossplane-workshop
      - <your-pair-id>
  providerConfigRef:
    name: default
  writeConnectionSecretToRef:
    name: <your-pair-id>-mysql-conn
    namespace: default
EOF
```

A few things worth understanding about this manifest:

- `metadata.name` and `forProvider.name` both start with your pair id. Kyverno's name-prefix policy denies any Aruba MR whose name doesn't carry your prefix — try removing the prefix on the next apply to see the rejection.
- `flavor: DBO2A4` is 2 CPU / 4Gi RAM, the smallest workshop tier. `DBO16A32` and friends are denied at admission.
- `engineId: mysql-8.0` and `location: ITBG-Bergamo` are the only allowed values for this workshop. The policy is a Helm value; your instructor can relax it via PR.
- `writeConnectionSecretToRef` tells Crossplane where to publish the connection details (host, port, admin credentials) when Aruba returns them. You'll read this Secret in §7.4.

### 3. Watch it provision

DBaaS provisioning takes about six minutes — Aruba is wiring up MySQL, networking, and storage. Open a watch:

```bash
kubectl get dbaas <your-pair-id>-mysql -w
```

Expected output once it's ready:

```
NAME                   SYNCED   READY   EXTERNAL-NAME
<your-pair-id>-mysql   True     True    69ee...
```

`SYNCED=True` means the Crossplane-to-Aruba API call landed; `READY=True` means Aruba says the instance is up and serving traffic. Press `Ctrl+C` once both are `True`.

<ValidateCheck check="aruba-mr-ready" />

## 7.4 Connect with `mysql` ⏱️ 10m

You now have a running MySQL instance — but Crossplane can't tell you where it lives or hand you a login. The Aruba provider doesn't yet publish the DBaaS endpoint, port, or admin credentials to the MR's connection Secret (the secret is created, but it's empty). That's a provider limitation, not a workshop one — see the "Under the hood" callout at the end of this section.

So this step is three small manual hops: read the endpoint from Aruba's console, create a database + user on the DBaaS, then build the connection Secret yourself.

### 1. Read the endpoint

Open your DBaaS in the Aruba console (your instructor will share the link). The detail view has a **Networking** tab — copy the private endpoint shown there. Standard MySQL port is `3306`.

For the rest of this section, we'll call those values `<endpoint>` and `3306`.

### 2. Create a database + user

Click the **Users and database** tab in the same DBaaS detail view. Use the two buttons:

- **Create database** → name it `appdb` (or anything; remember what you chose).
- **Create user** → username `appuser`, set a password Aruba accepts (typically: 8+ chars, upper + lower + digit + special). Remember the password.

Aruba enforces the password rules server-side; if your first pick is rejected, try something more conservative (e.g. mixed alphanumerics plus a single `!`).

### 3. Build the connection Secret

Stitch the four values into a Kubernetes Secret your Pod can read:

```bash
kubectl create secret generic <your-pair-id>-mysql-conn \
  --from-literal=endpoint='<endpoint>' \
  --from-literal=port='3306' \
  --from-literal=username='appuser' \
  --from-literal=password='<the-password>' \
  --from-literal=database='appdb'
```

### 4. Run a one-shot Pod that pings the DB

```bash
kubectl apply -f - <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: <your-pair-id>-db-test
spec:
  restartPolicy: Never
  containers:
    - name: mysql-client
      image: mysql:8
      command: ["sh", "-c"]
      args:
        - mysql -h "$endpoint" -P "$port" -u "$username" -p"$password" -e 'SELECT 1 AS ok;'
      envFrom:
        - secretRef:
            name: <your-pair-id>-mysql-conn
EOF
```

`envFrom: secretRef` loads every key from the connection Secret as an environment variable with the key name. The shell command then references each by name.

When the Pod completes:

```bash
kubectl logs <your-pair-id>-db-test
```

Expected output:

```
ok
1
```

You've routed a SQL query through `kubectl apply` of a Crossplane MR, into a real Aruba MySQL instance, back through your workshop cluster's network, and out the other side.

:::note Under the hood

In a mature provider you'd skip steps 1–3 entirely: the MR would write `endpoint` / `port` / `username` / `password` straight to the connection Secret, and `envFrom` would just work. `provider-arubacloud` doesn't yet — the embedded Terraform provider doesn't return those attributes from the create call, so there's nothing for Crossplane to publish.

There's an equally-Crossplane-native version of steps 2–3 using three more MRs (`Database`, `DBaaSUser`, `DatabaseGrant`) that we'd showcase if the provider supported cross-resource references for them; right now those MR types require literal Aruba IDs, so they trade one form of manual paste for another. Tracked in the provider's issue queue; once it lands, this section collapses back to the one-step version.

:::

## 7.5 Clean up ⏱️ 3m

Aruba bills the DBaaS hourly. Delete it now:

```bash
kubectl delete pod <your-pair-id>-db-test
kubectl delete dbaas <your-pair-id>-mysql
```

Crossplane reconciles the delete against Aruba, which tears down the MySQL instance. The connection Secret is owned by the DBaaS MR and is reaped along with it.

If you forget, your instructor sweeps up everything at workshop teardown — but the DBaaS will keep accruing hours until then.

## 7.6 What just happened

A YAML file in your cluster spun up a real MySQL database in Aruba's cloud — a managed instance with its own endpoint, ready for an app to connect to. You didn't open a vendor portal, you didn't paste a credit card, you didn't even see the API key. From your seat it looks exactly like creating a Kubernetes resource: write YAML, apply, wait for Ready. From Aruba's side, a real database is humming away on real hardware.

Same `Provider` → `ProviderConfig` → MR shape as AWS / GCP / Azure, applied against Aruba. The interesting differences:

- **The credential is platform-managed.** You never saw the Aruba API key; the platform team injected it into your cluster securely before you logged in. This is the production pattern for cloud-account isolation — developers request resources without ever holding the credential.
- **Kyverno guardrails at admission time.** Try editing the DBaaS YAML to use `flavor: DBO16A32`, or rename it without the `<your-pair-id>-` prefix, and re-apply. The error message comes back instantly — Kyverno saw the request before it touched etcd. The platform team can ship policy updates by PR; everyone's cluster picks them up on the next sync.
- **Aruba's network is more explicit than the hyperscalers.** AWS, GCP, and Azure all expose default networks; Aruba requires you reference VPC + subnet + security group URIs directly. The platform team manages those once for the workshop, then shares the URIs.

### Go deeper

- **Replace the §7.4 console steps with Crossplane MRs.** The user-and-database you clicked together in the Aruba console have native MR equivalents: `Database.database.arubacloud.crossplane.io` (logical schema), `DBaaSUser` (non-admin login), `DatabaseGrant` (the `GRANT … TO …` row that wires them together). Same `Provider` → API contract as the `DBaaS` MR you already wrote; you just need the parent DBaaS's Aruba ID from `kubectl get dbaas <your-pair-id>-mysql -o jsonpath='{.status.atProvider.id}'`. A working three-MR sequence (you fill in `<DBAAS_ID>` and `<PROJECT_ID>` from the workshop ConfigMap):

  ```yaml
  apiVersion: v1
  kind: Secret
  metadata:
    name: appuser-pw
  stringData:
    password: <pick-one-Aruba-accepts>
  ---
  apiVersion: database.arubacloud.crossplane.io/v1alpha1
  kind: Database
  metadata:
    name: appdb
  spec:
    forProvider:
      name: appdb
      dbaasId: <DBAAS_ID>
      projectId: <PROJECT_ID>
    providerConfigRef: { name: default }
  ---
  apiVersion: dbaasuser.arubacloud.crossplane.io/v1alpha1
  kind: DBaaSUser
  metadata:
    name: appuser
  spec:
    forProvider:
      username: appuser
      dbaasId: <DBAAS_ID>
      projectId: <PROJECT_ID>
      passwordSecretRef:
        name: appuser-pw
        namespace: default
        key: password
    providerConfigRef: { name: default }
  ```

  Apply that, wait for `Database/appdb` and `DBaaSUser/appuser` to be `Ready=True`, capture each MR's `.status.atProvider.id`, and `kubectl apply` the `DatabaseGrant` last:

  ```yaml
  apiVersion: databasegrant.arubacloud.crossplane.io/v1alpha1
  kind: DatabaseGrant
  metadata:
    name: appuser-on-appdb
  spec:
    forProvider:
      projectId: <PROJECT_ID>
      dbaasId: <DBAAS_ID>
      database: <DATABASE_ID-from-status.atProvider.id>
      userId: <DBAASUSER_ID-from-status.atProvider.id>
      role: readwrite
    providerConfigRef: { name: default }
  ```

  Two caveats — both upstream-fixable, both annoying right now:

  1. **No `*Ref` cross-references** on these MR Kinds yet, so the dbaasId, database, and userId fields are literal Aruba IDs you paste between steps. When the provider grows `dbaasIdRef` / `databaseRef` / `userIdRef`, those three lines become declarative cross-MR pointers and the order-of-apply pain disappears.
  2. **Aruba password rules are server-side.** First rejection is usually fixable by trying 8+ chars with upper + lower + digit + a single safe special; check the console's "Create user" tooltip if you're not sure.

  Once both are fixed, wrap all four MRs in an `AppDatabase` XR and platform users get a one-line "give me a DB for app X" workflow — same shape as the [`Application`](../crossplane-101/define-application) you wrote in 101 module 4.
