## 3.2 Installa la Helm chart вҸұпёҸ 7m

### 1. Aggiungi il repo Helm di Upbound

```bash

helm repo update upbound-stable
```

### 2. Installa UXP in `crossplane-system`

```bash
helm install crossplane upbound-stable/crossplane \
  --version 2.2.0-up.5 \
  --namespace crossplane-system \
  --create-namespace \
  --devel \
  --wait \
  --set webui.enabled=true
```

## 3.4 Cosa ГЁ appena successo

Un singolo comando Helm ha aggiunto un piccolo programma вҖ” Crossplane вҖ” al tuo cluster. Dal tuo punto di vista, il cluster ora capisce qualche parola nuova oltre a quelle Kubernetes built-in (`Deployment`, `Service`, `ConfigMap`, вҖҰ). ГҲ in ascolto, ma nessuno ha ancora pronunciato nessuna delle parole nuove, quindi non succede nulla di visibile.

Hai installato un controller Kubernetes. I cinque Pod in `crossplane-system` sono il controller core di Crossplane piГ№ i suoi compagni; insieme sono un programma long-running che osserva certi *kind* вҖ” `Provider`, `Function`, `CompositeResourceDefinition` (XRD), `Composition`, e qualunque kind composite-resource (XR) tu definisca вҖ” ed esegue un loop di riconciliazione su ognuno.

Ora Crossplane sta osservando questi cinque kind built-in ma non esistono ancora istanze, quindi il loop non ha nulla da fare. Nei prossimi quattro moduli gli darai lavoro:

- **Modulo 4** вҖ” applica una `Function` (la logica pluggable di una Composition) e il tuo primo `XRD` + `Composition` + `XR`. Crossplane reagirГ  creando una ConfigMap.
- **Modulo 5** вҖ” un XRD + Composition piГ№ grande che si dirama in un frontend + backend.
- **Modulo 6** вҖ” modifica la Composition; guarda il cambiamento propagarsi.
- **Modulo 7** вҖ” installa un `Provider` e usalo per gestire una release Helm.
