# Crossplane 2XX — Panoramica

Task choose-your-own-adventure per le coppie che hanno finito Crossplane 101. Un mix di walkthrough guidati ed esplorazioni open-ended — scegli il modulo che corrisponde a ciò che vuoi imparare dopo.

## Moduli disponibili

| Modulo | Forma | Argomento |
|---|---|---|
| [Crossplane Operations](./operations) | spunti open-ended | Pipeline imperative: `Operation` (one-shot), `CronOperation` (schedulata), `WatchOperation` (event-driven). Quando il loop dichiarativo non è lo strumento giusto — backup, rotazioni, fan-out, drift detection. |
| [(Guidato) Usa la CLI Crossplane](./crossplane-cli) | guidato, ⏱️ 23m | Anteprima, trace e validazione delle Composition con il binario `crossplane` — `render`, `beta trace`, `beta validate`, più puntatori ai comandi di authoring `xpkg`. |
| [Pattern di setup di un Provider](./provider-setup-pattern) | riferimento | La forma che ogni Provider segue: install + ProviderConfig + MR. Copre `ClusterProviderConfig` vs `ProviderConfig`, auth via Secret vs cloud-native, pinning della versione e famiglie di provider. Da leggere prima dei moduli cloud-provider [3xx](../crossplane-3xx/overview). |
| [Componi con altre function](./composition-functions) | spunti open-ended | Riscrivi una Composition del 101 con `function-go-templating`, `function-kcl` o `function-python` invece di `function-patch-and-transform`. Quando le patch smettono di scalare — condizionali su molte risorse, loop, codice che altrimenti sarebbe una catena di transform. |
| [(Guidato) Testa la tua Composition localmente](./testing-compositions) | guidato, ⏱️ 20m | Trasforma `crossplane render` in un loop di test pronto per la CI: cattura un render golden, fai il diff dopo una modifica alla Composition, alimenta lo stato osservato del cluster con `--observed-resources`. Si basa sul modulo CLI. |

Altri task atterrano qui mentre vengono scritti. Roadmap della sezione (per la [style guide](../) — il documento vivo della forma del workshop): provider extra, publishing OCI, status function e adozione di risorse esistenti in-cluster via `provider-kubernetes` (tracciato nella [issue #64](https://github.com/ricCap/crossplane-workshop/issues/64)).
