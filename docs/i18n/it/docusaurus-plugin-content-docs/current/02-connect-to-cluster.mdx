## 2.1 Prima di iniziare тП▒я╕П 2m

`kubectl` parla con *un* cluster тАФ quello descritto dal **kubeconfig** a cui la tua shell punta in questo momento. Per i prossimi cinque moduli, ogni comando deve atterrare nel tuo cluster del workshop, non in qualche altro cluster che potresti aver configurato sulla tua macchina.

Un kubeconfig ├и semplicemente un file YAML con l'indirizzo del cluster, un certificato e le credenziali. Stai per scaricare il tuo, dire alla shell di usarlo, e dimostrare che funziona applicando un manifest Pod di una sola riga.

## 2.2 Carica il tuo kubeconfig тП▒я╕П 5m

### 1. Scarica il file kubeconfig

Il tuo istruttore ti ha mostrato un QR code sulla board Miro. Inquadralo dal laptop (oppure clicca sul link che l'istruttore condivide) e scarica `<pair-id>.zip` тАФ il kubeconfig ├и racchiuso in uno ZIP protetto da password, cos├м un link leakato da solo non basta a entrare nel cluster.

### 2. Estrai con la password condivisa dall'operatore

L'istruttore condivide la password dello ZIP a voce o sulla board Miro dal vivo. Estrai il file YAML del kubeconfig dall'archivio тАФ la password ├и la stessa per tutte le coppie.

- **macOS / Linux:**
  ```bash
  cd ~/Downloads
  unzip <pair-id>.zip
  # chiede la password, poi scrive <pair-id>.yaml accanto allo zip
  ```
- **Windows:** click destro su `<pair-id>.zip` тЖТ **Estrai tuttoтАж** тЖТ inserisci la password condivisa dall'operatore тЖТ conferma. Otterrai `<pair-id>.yaml` nella stessa cartella.

### 3. Punta la tua shell al file

```bash
export KUBECONFIG=$PWD/<pair-id>.yaml
kubectl config current-context
```

(Su Windows / PowerShell: `$env:KUBECONFIG = "$PWD\<pair-id>.yaml"`.)

Output atteso:

```
workshop-<pair-id>
```

Il prefisso `workshop-` ├и il tuo segnale visivo che sei sul cluster giusto. Se vedi `docker-desktop`, `minikube`, `k3d-тАж` o qualsiasi cosa che non inizi con `workshop-`, l'`export` non ha avuto effetto in questa shell тАФ rieseguilo nello stesso terminale in cui lancerai il resto del workshop. Se `kubectl` stampa un errore, ricontrolla il percorso del file e che l'estrazione abbia effettivamente prodotto il YAML.

:::tip L'`export` vive solo in questo terminale
Apri una nuova tab o finestra e la variabile sparisce тАФ vedrai il tuo context *originale* (spesso `docker-desktop`) e qualunque comando `kubectl` colpir├а il cluster sbagliato. Tre modi per gestirla, dal pi├╣ semplice in poi:

1. **Rilancia l'export.** L'opzione pi├╣ economica per un workshop di 3 ore. Tieni la doc aperta e copia-incolla la riga in ogni nuovo terminale che apri.
2. **Persisti per il workshop (nessun altro cluster sul laptop).** Sposta il file in un posto pulito e fai puntare una volta il tuo profilo di shell:
   ```bash
   mkdir -p ~/.kube
   mv <pair-id>.yaml ~/.kube/workshop.yaml
   echo 'export KUBECONFIG=$HOME/.kube/workshop.yaml' >> ~/.zshrc   # oppure ~/.bashrc
   ```
   Apri un nuovo terminale тАФ `kubectl config current-context` dovrebbe stampare `workshop-<pair-id>` senza alcun export manuale. Rimuovi la riga da `~/.zshrc` ed elimina `~/.kube/workshop.yaml` a workshop finito.
3. **Sei un utente `kubectl` quotidiano con un `~/.kube/config` gi├а popolato?** Non sovrascriverlo тАФ distruggerebbe in silenzio ogni altro context che hai. *Fai il merge* del file del workshop:
   ```bash
   KUBECONFIG=$HOME/.kube/config:$PWD/<pair-id>.yaml \
     kubectl config view --flatten --raw > ~/.kube/config.new \
     && mv ~/.kube/config.new ~/.kube/config \
     && chmod 600 ~/.kube/config \
     && kubectl config use-context workshop-<pair-id>
   ```
   `--raw` mantiene intatto il token (e gli eventuali dati certificato); `--flatten` inlina i riferimenti a file; la catena `&&` evita che un `view` fallito sostituisca il tuo config vero con un file a met├а; `chmod 600` perch├й il file mergiato ora contiene il bearer token. Ora `kubectx` elenca `workshop-<pair-id>` accanto ai tuoi context abituali e puoi switchare. A workshop finito, pulisci:
   ```bash
   kubectl config delete-context workshop-<pair-id>
   kubectl config delete-cluster workshop-<pair-id>
   kubectl config delete-user    workshop-<pair-id>
   ```
:::

## 2.3 Smoke test тАФ applica un pod `hello` тП▒я╕П 3m {#23-smoke-test-apply-a-hello-pod}

Applica un Pod a un container cos├м il validator ha qualcosa di concreto su cui confermare che la tua pipeline `kubectl apply` scriva davvero nel cluster.

```bash
kubectl apply -f - <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: hello
  namespace: default
spec:
  containers:
    - name: hello
      image: public.ecr.aws/docker/library/busybox:1.36
      command: ["sh", "-c", "echo hello && sleep 3600"]
EOF
```

Aspetta qualche secondo per il pull dell'immagine, poi conferma:

```bash
kubectl get pod hello -n default
```

Output atteso:

```
NAME    READY   STATUS    RESTARTS   AGE
hello   1/1     Running   0          15s
```
