## 2.1 Prima di iniziare ⏱️ 2m

`kubectl` parla con *un* cluster — quello descritto dal **kubeconfig** a cui la tua shell punta in questo momento. Per i prossimi cinque moduli, ogni comando deve atterrare nel tuo cluster del workshop, non in qualche altro cluster che potresti aver configurato sulla tua macchina.

Un kubeconfig è semplicemente un file YAML con l'indirizzo del cluster, un certificato e le credenziali. Stai per scaricare il tuo, dire alla shell di usarlo, e dimostrare che funziona applicando un manifest Pod di una sola riga.

Stai per: scaricare un kubeconfig, esportarlo e applicare un pod `hello`.

## 2.2 Carica il tuo kubeconfig ⏱️ 5m

### 1. Scarica il file kubeconfig

Il tuo istruttore ti ha mostrato un QR code sulla board Miro. Inquadralo dal laptop (oppure clicca sul link che l'istruttore condivide) e scarica `<pair-id>.zip` — il kubeconfig è racchiuso in uno ZIP protetto da password, così un link leakato da solo non basta a entrare nel cluster.

### 2. Estrai con la password condivisa dall'operatore

L'istruttore condivide la password dello ZIP a voce o sulla board Miro dal vivo. Estrai il file YAML del kubeconfig dall'archivio — la password è la stessa per tutte le coppie.

- **macOS / Linux:**
  ```bash
  cd ~/Downloads
  unzip <pair-id>.zip
  # chiede la password, poi scrive <pair-id>.yaml accanto allo zip
  ```
- **Windows:** click destro su `<pair-id>.zip` → **Estrai tutto…** → inserisci la password condivisa dall'operatore → conferma. Otterrai `<pair-id>.yaml` nella stessa cartella.

### 3. Punta la tua shell al file

```bash
export KUBECONFIG=$PWD/<pair-id>.yaml
kubectl config current-context
```

(Su Windows / PowerShell: `$env:KUBECONFIG = "$PWD\<pair-id>.yaml"`.)

Output atteso:

```
workshop-<pair-id>
```

Il prefisso `workshop-` è il tuo segnale visivo che sei sul cluster giusto. Se vedi `docker-desktop`, `minikube`, `k3d-…` o qualsiasi cosa che non inizi con `workshop-`, l'`export` non ha avuto effetto in questa shell — rieseguilo nello stesso terminale in cui lancerai il resto del workshop. Se `kubectl` stampa un errore, ricontrolla il percorso del file e che l'estrazione abbia effettivamente prodotto il YAML.

:::tip L'`export` vive solo in questo terminale
Apri una nuova tab o finestra e la variabile sparisce — vedrai il tuo context *originale* (spesso `docker-desktop`) e qualunque comando `kubectl` colpirà il cluster sbagliato. Tre modi per gestirla, dal più semplice in poi:

1. **Rilancia l'export.** L'opzione più economica per un workshop di 3 ore. Tieni la doc aperta e copia-incolla la riga in ogni nuovo terminale che apri.
2. **Persisti per il workshop (nessun altro cluster sul laptop).** Sposta il file in un posto pulito e fai puntare una volta il tuo profilo di shell:
   ```bash
   mkdir -p ~/.kube
   mv <pair-id>.yaml ~/.kube/workshop.yaml
   echo 'export KUBECONFIG=$HOME/.kube/workshop.yaml' >> ~/.zshrc   # oppure ~/.bashrc
   ```
   Apri un nuovo terminale — `kubectl config current-context` dovrebbe stampare `workshop-<pair-id>` senza alcun export manuale. Rimuovi la riga da `~/.zshrc` ed elimina `~/.kube/workshop.yaml` a workshop finito.
3. **Sei un utente `kubectl` quotidiano con un `~/.kube/config` già popolato?** Non sovrascriverlo — distruggerebbe in silenzio ogni altro context che hai. *Fai il merge* del file del workshop:
   ```bash
   KUBECONFIG=$HOME/.kube/config:$PWD/<pair-id>.yaml \
     kubectl config view --flatten > ~/.kube/config.new
   mv ~/.kube/config.new ~/.kube/config
   kubectl config use-context workshop-<pair-id>
   ```
   Ora `kubectx` elenca `workshop-<pair-id>` accanto ai tuoi context abituali e puoi switchare. A workshop finito, pulisci con `kubectl config delete-context workshop-<pair-id>` (più le corrispondenti `delete-cluster` e `delete-user`).
:::

### 4. Elenca i namespace come smoke test

```bash
kubectl get namespaces
```

Dovresti vedere i namespace di sistema standard (`default`, `kube-system`, `kube-public`, …). Se vedi un errore di permessi, controlla di stare usando il kubeconfig della tua coppia — ogni coppia ha accesso solo al proprio cluster.

### 5. (Opzionale) Installa qualche tool comodo

Non sono richiesti, ma renderanno il resto del workshop più rapido. Salta se sei stretto col tempo.

```bash
brew install kubectx k9s helm      # macOS / Linuxbrew
```

I link di installazione per Windows / Linux puro sono nel [Cheatsheet](./cheatsheet#12-tools-worth-installing).

### 6. (Fortemente consigliato) Mostra il cluster nel prompt della shell

Il modo più efficace per evitare di lanciare `kubectl apply` o `helm install` sul cluster sbagliato è rendere il context attivo visibile nel prompt della shell. Installa [`kube-ps1`](https://github.com/jonmosco/kube-ps1) (Bash/Zsh) oppure abilita il modulo [`kubernetes`](https://starship.rs/config/#kubernetes) di [Starship](https://starship.rs) — entrambi mostrano il context corrente accanto a ogni prompt. Quando vedi `(workshop-<pair-id>)` sei al sicuro; quando vedi `(docker-desktop)` sai di sistemare la cosa prima di digitare. Le indicazioni di installazione sono nel [Cheatsheet](./cheatsheet#12-tools-worth-installing).

## 2.3 Smoke test — applica un pod `hello` ⏱️ 3m {#23-smoke-test-apply-a-hello-pod}

Applica un Pod a un container così il validator ha qualcosa di concreto su cui confermare che la tua pipeline `kubectl apply` scriva davvero nel cluster.

```bash
kubectl apply -f - <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: hello
  namespace: default
spec:
  containers:
    - name: hello
      image: public.ecr.aws/docker/library/busybox:1.36
      command: ["sh", "-c", "echo hello && sleep 3600"]
EOF
```

Aspetta qualche secondo per il pull dell'immagine, poi conferma:

```bash
kubectl get pod hello -n default
```

Output atteso:

```
NAME    READY   STATUS    RESTARTS   AGE
hello   1/1     Running   0          15s
```

<ValidateCheck check="hello-pod" />

Quando la tile diventa verde, il tuo kubeconfig è a posto e `kubectl apply` funziona end-to-end.

## 2.4 Cosa è appena successo

Hai puntato il tuo terminale su un cluster Kubernetes remoto e gli hai chiesto di eseguire un piccolo container — e l'ha fatto. Il file kubeconfig è la rubrica: dice a `kubectl` con quale cluster parlare, chi sei tu e come autenticarti. Il Pod `hello` è il più piccolo "è acceso?" possibile — un container, niente rete, niente storage — e il fatto che sia arrivato in `Running` dimostra che tutta la catena funziona.

Hai detto alla tua shell con quale cluster parlare e hai dimostrato la connessione scrivendo una risorsa che il validator può rileggere. Tutto dal modulo 3 in poi assume che questo funzioni.

### Per approfondire

- [Documentazione di kubectl config](https://kubernetes.io/docs/reference/kubectl/generated/kubectl_config/) — tutto ciò che `kubectl config` può fare.
- [Riferimento del file kubeconfig](https://kubernetes.io/docs/concepts/configuration/organize-cluster-access-kubeconfig/) — cosa c'è dentro lo YAML che hai appena scaricato.
- Puoi lasciare il pod `hello` in esecuzione — non usa risorse significative e i moduli successivi lo ignorano. Se preferisci ripartire pulito: `kubectl delete pod hello -n default`.
